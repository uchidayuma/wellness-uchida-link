最終更新: 2026-05-16

## 基本方針

**「あなたの課題、わかります。解決できます。」を順番に証明していく。**

- 技術スタックの羅列ではなく、顧客が得る成果で語る
- 顧客の感情・課題から入る（自分のスキルから入らない）
- 価格はサービスページのみ。トップページには出さない
- CTAは低圧力。「まず状況を聞かせてください」スタイル
- 週3日稼働は顧客に関係ないので表に出さない

## 各セクション詳細

### 1. Hero

**キャッチコピー案：**
```
社内に詳しい人がいなくても、
インフラは仕組みで回せます。
```

または：
```
インフラが属人化している。
品質が安定しない。
それは、仕組みがないからです。
```

**自己紹介文（実績を自然に含める）：**
```
Docker・AWS ECS/Fargate・Kubernetes を専門とするインフラエンジニアです。
コンテナ化・IaC・CI/CD の構築から、チームの開発プロセス整備まで担います。
Udemyでは2.6万人にコンテナ技術を教えてきた経験から、
現場で躓くポイントと、それを防ぐ仕組みを熟知しています。
```

**実績バッジ（3つ）：**
- Udemy 2.6万人受講生（Udemyページへリンク）
- Best Seller 認定（Udemyページへリンク）
- インプレス研修 満足度95%+（インプレスページへリンク）

**CTA：**
- 主：「まず30分、現状を聞かせてください」→ Calendly
- 副：レジュメ / GitHub

**CTA周辺の一言（低圧力）：**
```
費用や進め方の話はその後で構いません。
```

### 2. 3本柱

各柱の構成：
```
[問いかけ（顧客の課題・感情）]
  ↓
[解決：何をするか]
  ↓
[成果：数値・事実]
  ↓
[詳細ページへのリンク]
```

| 柱 | 問いかけ | 解決 | 成果例 | リンク先 |
|----|---------|------|--------|---------|
| ① コンテナ化・IaC | 「サーバーの管理に、人手が取られていませんか？」 | インフラをコードで管理し、手作業をなくす | デプロイ時間 50% 削減 | infrastructure.html |
| ② CI/CD・自動テスト | 「リリースのたびに、バグが怖くなっていませんか？」 | 自動化パイプラインで品質と速度を両立 | バグ発生率 6% → 3% | cicd.html |
| ③ チーム仕組み化 | 「あなたがいないと、開発が止まりますか？」 | 開発ルールとプロセスを整備し、属人化を解消 | 管理者不在でも開発継続 | management.html |
